Definition

What this is.

One home for everything a taste-maker saves across Pinterest, Arena, Cosmos, Instagram, LinkedIn, Substack, and the open web — searchable, cluster-able, and owned by the user.

  • Memory layer — every save becomes context the rest of the Sun suite can act on.
  • Taste artifact — a living record of what the user finds good, not what an algorithm pushes.
  • Search-first — conversational queries across everything saved, across every source.
  • Owned — the user exports, deletes, or hands their taste to another app they trust.
Problem

Why this exists.

Taste-makers are saving constantly — and losing it constantly. Current tools hold fragments of a person's taste; no tool holds the whole.

  • Scatter. Saves live across Pinterest, Instagram DMs, LinkedIn, Chrome bookmarks, Substack, iMessage. None of it is one library.
  • Stale AI. Claude pitches 2023-era references because it has no visibility into what the user is actually saving now.
  • Lost ownership. Instagram, TikTok, LinkedIn own the signal. The user doesn't get paid for their taste; the platforms do.
  • No memory. Every new design brief starts from zero. References are re-found, not recalled.
Vision

Where we're going.

Sun-Pull becomes the memory layer for taste on the internet. The cool-kids-club of curators — earned by saving, not bought.

  • Own your data. Every save is portable, exportable, deletable. The user, not the platform, holds the library.
  • Curation is currency. The more rare and specific your saves, the more rare and specific the pool you have access to.
  • Collaborative filtering. Users with adjacent taste auto-share into each other's libraries — optionally — so the pool grows ambiently.
  • Sign in with Sun-Pull. Any app can ask for your taste at sign-in; homepages land pre-curated. Users and Sun-Pull share the revenue.
  • From B2B to consumer. Start with taste-making designers; extend to travel, food, music, film — anywhere taste lives.
Ecosystem

Where this sits.

Sun-Pull is the memory; Sun-Run is the action; Sun-Dial is the calendar; Sun-Chat is the interface. Sun-Pull is the substrate — built and shipped first because the others depend on it.

Sun-Dial

Sun-Chat

Personas

Who we're designing for.

Phase 1 is narrow by design. The flows only need to serve one persona well before we widen the aperture.

Archetype
High-agency taste-maker. Designer, founder, or strategist whose job depends on being early.
Behaviour
Saves constantly — Pinterest, Instagram DMs, LinkedIn, bookmarks, Substack. Already has years of back-catalogue scattered.
Jobs to be done
(1) Stop losing saves. (2) Find old saves by vibe, not by filename. (3) Give AI tools the right references on the first try.
What they already buy
Notion, Are.na, Mymind, Raindrop, Readwise, Pinterest Premium.
Success signal
They stop saving anywhere else. Sun-Pull becomes the default verb: "I'll pull that."
Mental model

What this feels like.

Pinterest masonry for the grid; a lightweight OS for the chrome around it. Dense, visual, owned. Not a database. Not a feed.

  • Pinterest, for the grid. Masonry, image-first, scan-able at speed.
  • OS, for the shell. Windows, cluster chips, a searchable command layer — a desktop, not a webpage.
  • Private, by default. The library is the user's. Sharing is explicit, per-cluster.
Wireframe → masonry grid with cluster chips and an OS-style top chrome.
Vocabulary

The words we use.

A small, tight vocabulary. Every surface uses these words consistently.

Save
A single item — image, link, post, file — added to the library from any source.
Source
The origin of a save (Pinterest, Instagram, LinkedIn, upload, text-to-pull, Cmd+S).
Cluster
An auto- or manually-grouped set of saves sharing intent ("consumer launches", "brand prospects").
Library
The user's full set of saves — home surface.
Archetype
A read on the user's taste pattern, surfaced after onboarding.
Pull
Verb. To save something to Sun-Pull. "I'll pull that."
Principles

Rules we design against.

Every decision traces back to one of these. If we can't trace it, we're drifting.

  • Friction-less save. If saving takes a second thought, we've already lost.
  • Text is enough. On phone, an SMS is a valid interface. No app required.
  • Visual by default. Every save gets an image the user can scan past.
  • The library is the user's. Export, delete, disconnect — all one click.
  • Silence over nudge. We don't chase engagement. No streaks. No shame.
  • Clusters emerge. The user doesn't tag — the system proposes and they confirm.
Journey

The arc.

Five moments from first contact to steady state. Each should feel inevitable — not sold to.

  1. Discover. A designer hears Sun-Pull is where their peer group holds their saves.
  2. Onboard. Connect accounts + import bookmarks. One loading screen, one archetype reveal.
  3. Land. Library is already populated. First search works on day one.
  4. Habit. Text-to-pull and Cmd+S replace every other save action within two weeks.
  5. Retention. A weekly "your taste this week" card; archetype drift tracked over time.
Auth

Sign in, connect, disconnect.

Phase 1 is internal — invite-only email sign-in. But the model we set now has to support the "Sign in with Sun-Pull" endgame.

  • Sign in. Email magic-link for phase 1. OAuth (Google, Apple) for phase 2.
  • Connect sources. Per-source OAuth where available (Pinterest, LinkedIn via extension). Explicit scope list shown at each connect.
  • Disconnect. One click per source. Disconnect preserves saves already pulled; revokes future ingestion.
  • Failure states. Expired token → re-auth prompt in the source's row, not a modal. Rate-limit → deferred re-sync, no user error shown.
  • Future: "Sign in with Sun-Pull" — third-party apps request a scoped taste read; user approves per-request.
Flow

First visit.

Four beats — input, load, archetype reveal, land on library. Every beat has a single decision. Drop-off risk is the load screen if import stalls.

  1. 1 · Input. Connect two sources (min), or drop a links.txt / bookmarks.html. Skip allowed — empty library is a valid state.
  2. 2 · Loading. Live counter of items pulled. Estimated completion. Option to continue in background.
  3. 3 · Archetype. A single card — "You're a ___ saver." Two-sentence explanation. Shareable.
  4. 4 · Library. Land on the populated grid with a brief coach-mark on search, cluster chips, and the pull verb.
Wireframe → 4-step strip, mobile + desktop.
Flow

Return visit.

The return visit is the library — no welcome screen, no streak. A "since last visit" row is the only acknowledgment of return.

  • Land directly on Library, scrolled to top.
  • A single "Since you were here" row shows new auto-pulls and new cluster proposals.
  • Cluster proposals are dismissible inline — no modal.
  • No onboarding re-prompts; source errors live in Settings, not the home surface.
Wireframe → library with a "since last visit" strip at the top.
Artifact

The library.

The single headline surface. Pinterest-dense masonry, OS-style chrome, a command bar that is also the search. Everything else is accessed from here.

  • Command bar. Top of screen. Conversational search doubles as a jump-to for clusters and settings.
  • Cluster chips. Horizontal row of auto-proposed and user-confirmed clusters. Click to scope the grid.
  • Masonry grid. Image-first cards. Hover reveals source logo + date. Click opens the detail panel.
  • Source rail. Thin left rail showing connected sources and their sync state.
Wireframe → command bar · cluster chips · masonry · source rail.
Surface

A save, up close.

A right-side panel, not a modal. Shows the save's metadata and every Run or project it's powering. The link back to Sun-Run lives here — the only place the two products touch in phase 1.

  • Metadata. Source, captured-at, original URL, any note the user added.
  • Clusters. Every cluster the save belongs to, inline-removable.
  • Used by. Runs and projects sourcing this save (phase 2 link; phase 1 shows placeholders).
  • Actions. Note, delete, share to a cluster, export.
Wireframe → right panel with image, metadata, clusters, "used by" list.
Case

Conversational search.

The user types in sentences, not keywords. Filters are chips that appear as the query resolves — never a separate panel.

  • Query. "Consumer launches I saved last year." Resolves across every source.
  • Facets as chips. Source, date range, cluster — all appear under the query, editable inline.
  • Results. Masonry with a "result-count" chip; empty state proposes related queries.
  • Save the search. A saved search becomes a virtual cluster.
Wireframe → search bar with resolved chips + results grid.
Case

Clusters emerge.

The user never tags. The system proposes clusters from behaviour and asks one yes/no. A cluster can also be a saved search.

  • Auto-propose. When a cohort of saves share visual/semantic signal, Sun-Pull proposes a cluster name and the user confirms, renames, or dismisses.
  • Manual. Select saves → "new cluster" → name. No mandatory description.
  • Nesting. Flat in phase 1. Nested clusters deferred to v2.
  • Shared clusters. Phase 2+. Share a cluster with another Sun-Pull user — read-only or contribute.
Wireframe → cluster chip row + a proposal card ("new cluster suggested").
Case

Bringing the back-catalogue.

Everyone onboarding has ten years of saves. The import flow has to feel like a one-time gift, not homework.

  • Pinterest / Arena / Cosmos. Direct API. Shows boards + saves; user picks all or specific boards.
  • LinkedIn. Chrome extension (LinkedIn has no save-API). Extension walks the user through "open your saves, we'll pull."
  • Chrome bookmarks. Drop a bookmarks.html file. Parsed server-side.
  • Files & links. Drag a folder or paste a links.txt. Each becomes a save.
  • Failure states. Partial import → retries surface as banner, not blocker. User can keep browsing while sync continues.
Wireframe → source picker + per-source connect state + drag-drop zone.
Case

Text to pull.

Saving on phone is a text message. No app open. The only interface is a thread with Sun-Pull.

  • Send. User texts a link + optional context to Sun-Pull.
  • Confirm. Sun-Pull replies with the preview and the auto-proposed cluster — reply "yes" to confirm, or type a different cluster.
  • Run hook (phase 2). If the save matches an active Run, Sun-Pull asks: "Add this to the brand-prospects Run?"
  • Failure states. Dead link → Sun-Pull asks for a screenshot. Private content → refuses and explains why.
Wireframe → SMS conversation, three bubbles.
Case

Cmd+S, anywhere.

A browser extension binds Cmd+S to a micro-popover — no tab-switch, no modal, no focus steal. Confirm or type a one-line context, and it's gone.

  • Trigger. Cmd+S on any page (optional keybind).
  • Popover. 320px overlay with the page preview, a cluster auto-suggestion, and a one-line context input.
  • Dismiss. Enter commits and closes; Esc cancels. Closes itself on commit — no toast, no confirmation modal.
  • Failure states. Auth-expired → extension shows a single-line "reconnect" CTA; save is queued until reconnect.
Wireframe → browser page with Cmd+S micro-popover overlay.
Integrations

What Sun-Pull feeds.

In phase 1 Sun-Pull stands alone. The slide exists so we design with these hooks in mind, not so we ship them.

Sun-Run
Runs read clusters as inputs. A save added with context can attach itself to an open Run.
Sun-Dial
Runs carry deadlines from Sun-Dial. Sun-Pull only needs to expose stable save IDs.
Sun-Chat
Every chat turn can pull from the library; every chat answer cites the save it leaned on.
Third-party
"Sign in with Sun-Pull" — scoped taste reads for any app the user trusts.
Settings

Trust, data, control.

The product's credibility lives here. If any of these are hard to find, the "you own your data" claim collapses.

  • Export. Full-library export as JSON + media zip. One click, no email.
  • Delete. Per-save, per-cluster, per-source, or full account — each reversible for 30 days.
  • Disconnect. Per-source OAuth revoke; existing saves stay.
  • Visibility. Private by default. Sharing is per-cluster and explicit.
  • Revenue share (phase 3). Opt-in. Dashboard shows what was shared, with whom, and the cut.
Edges

Quiet failures.

Never shame the user for an empty library or a broken sync. The tone stays dignified — descriptive, not apologetic.

  • Empty library. "Nothing pulled yet. Connect a source or text a link to get started."
  • Source disconnected. Inline row: "Pinterest disconnected on Apr 18. Reconnect."
  • Import stalled. Banner with live count, not a modal. Continue-in-background is always available.
  • Dead save. Original URL 404s → cached preview stays, with a small "original removed" tag.
  • Cluster proposal rejected. Dismissing a proposal never asks "are you sure"; the proposal just disappears.
  • Private content. If a save can't be captured (private Instagram), Sun-Pull refuses and shows the reason.

Questions

What to bring back.

Decisions design should lead on. Each question paired with what a "good answer" looks like.

  • Archetype taxonomy. How many archetypes? What are they named? — Good answer: a fixed set of 8–12 with distinct visuals.
  • Cluster proposal cadence. When does the system propose? Every N saves, or daily? — Good answer: a clear trigger that doesn't feel nagging.
  • Source prioritisation. Which sources ship in phase 1 import? — Good answer: the top three by user's save-volume.
  • Dedupe. If the same URL is saved twice across sources, is it one save or two? — Good answer: one, with both source tags.
  • Archetype drift. Does the archetype update over time, or is it a moment? — Good answer: a moment now, drift tracked in phase 2.
Metrics

How we'll know it worked.

Phase 1 is internal — metrics are behavioural, not volume. Good numbers look like habit formation, not virality.

  • Daily pull rate. Saves per active user per day. Good: >3 within week two.
  • Source concentration. % of a user's saves that land in Sun-Pull vs. elsewhere. Good: >70% by week four.
  • Search success. % of searches that end in a click. Good: >60%.
  • Cluster acceptance. % of auto-proposed clusters confirmed, not dismissed. Good: >40%.
  • Onboarding completion. % who reach the library from cover. Good: >80%.
  • Not optimizing for: DAU, session length, notification open-rate.
